PIOS Core Start

PIOS Core is personal information infrastructure for one owner. It preserves source material, records canonical events, maintains structured knowledge, and lets agents retrieve context without taking ownership away from the person.

Agent instruction: help the owner choose a path, ask only the questions needed for that path, and stop at every approval gate. Do not deploy infrastructure, upload owner data, start connector sync, migrate broad folders, or publish anything without explicit owner approval.

Choose Deployment

Core Managed AWS

Use this when the owner wants to deploy an AWS-native Core into the owner's own AWS environment using templates, manifests, and gated runbooks. This is not a hosted peecos account service.

Choose Adoption Track

Track 1: New Owner

Use this for a clean start. Keep first intake narrow, low sensitivity, and owner-approved.

https://github.com/peecos/pios-core-aws-template/blob/main/docs/runbooks/track-1-new-owner-gradual-start.md

Track 2: Existing System

Use this when the owner already has custom wikis, app folders, automation, agents, dashboards, logs, takeouts, or other personal information infrastructure.

https://github.com/peecos/pios-core-aws-template/blob/main/docs/runbooks/track-2-existing-system-transition-template.md

Required Owner Questions

  1. Which deployment option: AWS Managed or Self-Hosted VM?
  2. Which adoption track: Track 1 clean start or Track 2 existing system?
  3. What should the owner id, owner slug, and first environment name be?
  4. What data sensitivity is allowed for the first proof?
  5. Who can approve infrastructure, upload, migration, and decommissioning?
  6. For AWS: account id, region, operator principal, auditor principal, cost guardrail, and MFA posture.
  7. For self-hosted: local machine, VM host, or cloud VM provider; who controls the host; and whether provider import is already proven.

Fail-Closed Rules

Stop if sensitivity is unknown, guarded data may be involved, a command would deploy infrastructure, a command would upload owner data, a command would start connector sync, a command would start broad migration, a command would delete sources, a manifest sets authorization gates to true, a checksum or signature fails, or provider import is only experimental for the target.

Unknown sensitivity means exclude. An unresolved blocking question means fail.

Current Self-Hosted Status

  • Local QEMU image path has proof artifacts.
  • Release package extraction and boot proof passed.
  • Production release-key setup passed.
  • A signed v0.1.0-rc1 prerelease is published in the peecos GitHub organization.
  • Google Cloud is experimental, not supported. Azure and external VM providers are unverified.
  • AWS owners should use the AWS template setup path unless they specifically need a separate self-hosted EC2 proof.

Documentation Boundary

peecos.org is the public orientation layer. Full technical documentation, release verification instructions, runbooks, proof records, and master-level specifications belong in the peecos GitHub repositories.
